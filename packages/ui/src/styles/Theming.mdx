# Theming & Customization

WS-UIKit is styled entirely with **CSS custom properties** — no JS theming, no build
step, no provider. You customise it by setting variables in your own CSS. Switching a
whole look is just loading a different set of token values.

---

## The three tiers

Every visual value resolves down a chain, each tier overridable on its own:

```
component  →  --<comp>-*        (component token — per-component knob)
                → semantic       (--color-action-*, --radius-control, --gap-control …)
                   → primitive    (--palette-*, --space-N, --radius-N)
```

- **Primitives** — raw scale: `--space-4`, `--radius-8`, `--palette-primary-700`.
- **Semantic roles** — meaning: `--color-action-rest`, `--radius-control`, `--gap-control`.
  Retune a role once → every component that uses it follows.
- **Component tokens** — `--button-radius`, `--button-gap`. A curated per-component knob.
  Default falls through to a semantic role, so you rarely need them.

> `--<comp>-*` is the public, overridable API. `--_*` (underscore) is **private** —
> internal to a component, do not set it.

> **Raw values (hex, `rgb`, `px`, `rem`) live only in the primitive tier.** Semantic and
> component tokens always _reference_ a primitive (`var(--palette-*)`, `var(--space-*)`) —
> never a hardcoded literal. So you recolor/resize a brand by overriding **primitives**, and
> everything above follows.

---

## The three altitudes of override

Pick the **lowest** one that does the job.

### 1. Global — a brand / theme

Set semantic roles (or primitives) at the root. Affects the whole app, centrally.
The kit reads `data-theme` and `data-brand` on `<html>`:

```css
/* my-brand.css */
:root[data-brand="acme"] {
  /* retune the palette → every action color follows */
  --palette-primary-500: #6d28d9;
  --palette-primary-700: #5b21b6;

  /* or retune a semantic role directly */
  --radius-control: var(--radius-2);   /* all controls less round   */
  --gap-control:    var(--space-2);    /* tighter control spacing   */
}
```

```html
<html data-theme="dark" data-brand="acme">
```

Switching look = swapping which brand CSS is loaded (or flipping the attribute at runtime —
no re-render, works before hydration).

### 2. Zonal — a region of the page

Set a component token on a wrapper. Applies to everything inside it.
⚠️ CSS variables inherit — this affects **all** nested instances in the subtree.

```css
.toolbar {
  --button-radius: 0;          /* square buttons inside .toolbar */
  --button-gap: var(--space-2);
}
```

### 3. Instance — one element

Set the token inline (or via a className) on the single element:

```jsx
<MButton style={{ "--button-radius": "var(--radius-pill)" }}>Pill</MButton>
```

## What you can override

**Semantic roles** (global, shared across components):

| Group | Tokens |
|-------|--------|
| Action colors | `--color-action-rest` / `-hover` / `-active` / `-foreground` / `-disabled`, `--color-action-secondary-*`, `--color-action-tertiary-*`, `--color-action-subtle-*`, `--color-action-outlined-*`, `--color-action-transparent-*` |
| Text / surface | `--color-text-primary` / `-secondary` / `-inverse`, `--color-surface*`, `--color-border-*` |
| Radius roles | `--radius-control`, `--radius-control-sm`, `--radius-surface`, `--radius-pill` |
| Spacing roles | `--gap-control`, `--padding-control-x/-y`, `--padding-control-{x,y}-{s,m,l}` |
| Focus ring | `--focus-ring-width`, `--focus-ring-offset`, `--color-focus-ring` (uniform for all interactive elements) |

**Component tokens** — the per-component knobs are listed in each component's own
**Customization** section (see e.g. **Atoms/Form/MButton → Customization**).

---

## Cascade layers

The kit declares an explicit `@layer` order so overrides are predictable regardless of
selector specificity or import order:

```css
@layer primitives, semantic, brand, components, overrides;
```

Your app styles are unlayered, so they win over the kit's layers by default. If you need
your overrides to sit in a specific place, wrap them in `@layer overrides { … }`.
